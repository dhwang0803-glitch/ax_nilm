## [슬라이드 2] 문제 정의 — 왜 이 서비스가 필요한가?

**핵심 메시지:** 기존 에너지 절감 정책은 "결과 판정"만 합니다. 사용자에게 필요한 건 "과정 코칭"입니다.

**기존 KEPCO 주택용 에너지캐시백의 한계:**
- 직전 2개년 동월 평균 대비 3% 이상 절감했나 → 월말에 판정 후 지급/미달 통보
- 총 사용량(계량기 한 개)만 봄
- "얼마나 줄었나"만 알려줌 — "어떻게 줄일 수 있나"는 없음
- 고객은 결과를 수동으로 받아보는 구조

**사용자에게 실제로 필요한 것:**

| 필요 기능 | 설명 |
|-----------|------|
| 가전별 분해 (NILM) | 단일 계량기에서 에어컨·냉장고·세탁기 등 개별 소비량 파악 |
| 사전 예측 | 월말 전에 "지금 페이스면 캐시백 못 받는다"고 미리 알려줌 |
| 실행 가능한 권고 | "에어컨 설정온도 1도 올리면 캐시백 기준 달성 가능" 수준의 구체적 행동 제안 |
| 날씨 연동 | 다음 주 기온 오를 때 선제적으로 소비 조정 권고 |
| 누진 단계 관리 | 현재 누진 단계·남은 kWh 조회 → LLM 절감 권고에 반영 |
| 이상 탐지 | 가전 성능 저하나 대기전력 이상 등 자동 감지 |

## [슬라이드 4] NILM이란?

**NILM (Non-Intrusive Load Monitoring)**  
= 비침습 부하 모니터링

**핵심 아이디어:**
- 물결처럼 섞인 전체 전력 신호에서 각 가전의 "전력 지문"을 찾아낸다
- 냉장고 압축기가 켜지면 특유의 파형 → 학습된 모델이 식별

**성능 목표 — 선택과 집중 전략:**

| 가전 그룹 | 대표 기기 | F1 목표 | 근거 |
|-----------|-----------|---------|------|
| 핵심 에너지 관리 가전 | 냉장고, 에어컨, 세탁기 | ≥ 0.85 | 고전력·뚜렷한 파형 → 신호 식별 용이 |
| 저전력·상시 패턴 가전 | TV, 셋톱박스, 공유기 | 0.40 ~ 0.50 | 대형 가전 동작 시 신호 중첩으로 분리 한계 |
| **전체 22종 평균** | — | **~0.60** | 타겟 10종(~75%) 대비 22종 확장 시 학계 통계 기준 합리적 목표 |

- 학계 통계: 타겟 가전 5개 증가마다 F1 약 7%p 하락 → 10종(75%) → 22종 시 55~60% 달성은 고무적
- MAE: 핵심 가전 기준 **< 50W** (절대값 기준, 가전별 정격 전력 대비 비율은 의미 없음)


> **기술적 희소성:** 단일 계량기로 22종 동시 분리 — 핵심 가전 90% 육박, 전체 평균 60%

## [슬라이드 5] 핵심 기술 — CNN + TDA 하이브리드 모델

**왜 Seq/BERT가 아닌 CNN + TDA인가?**

| 비교 아키텍처 | NILM 환경에서의 치명적 약점 | 기각 및 채택 근거 |
|---|---|---|
| LSTM / Seq2Seq | 순차 처리로 인한 병렬화 한계 | 초당 수십 샘플(30Hz)이 들어오는 환경에서 요구 응답속도(< 500ms) 달성 불가 |
| BERT / Transformer | Self-attention의 O(n²) 메모리 복잡도 | 고주파 시계열 특성상 시퀀스가 길어 연산/메모리 비용 폭등, 엣지(Edge) 탑재 불가 |
| 순수 1D CNN | 국소 패턴 추출엔 강하나, 중첩에 취약 | 여러 가전이 켜져 파형이 겹치는 신호 가림(Signal Eclipse) 발생 시 오분류 급증 |
| **🚀 CNN + TDA (제안)** | 위상 지문 추출을 통한 신호 중첩 극복 | CNN이 기본 파형을 잡고, TDA가 겹친 파형을 3차원으로 찢어서 식별하는 역할 분담 |

> **결정 근거:** 실시간 NILM은 장기 의존성(Long-term dependency) 문제가 아니라, "짧은 과도 파형의 포착 + 겹친 신호의 구조적 분해" 문제입니다.  
> 따라서 무거운 시계열 모델 대신 국소 패턴에 강한 CNN과 위상 분석에 강한 TDA의 결합이 가장 압도적인 효율을 냅니다.

## [슬라이드 6] 이상 탐지 엔진 — 설계 & 탐지 로직

**핵심 설계 원칙:**
1. **오탐 최소화** — 통계 기반 임계값 탐지
2. **원인 설명성** — 이상 발생 시 수치 근거와 함께 이벤트 기록
3. **확장성** — 팀원 AI Agent가 DB 이벤트를 읽어 LLM 진단 리포트 생성

**ANOM-001: 통계 기반 성능 저하 감지**
- 소비량 증가: 현재 주간 평균 > 과거 3개월 평균 + 20%
- 작동시간 증가: 월간 작동시간 > 기준값 + 30%
- 피크 변화: 최대 전력 > 기준값 + 30%

> **[PoC 데모 한계]** AIHub 데이터셋 31일치 제한 → 베이스라인 3주 / 검증 1주 단축 적용. 실제 데이터 확보 시 임계값 재튜닝 예정.

**ANOM-002: 학습 기반 비정상 탐지 (기획)**
- Isolation Forest — 다변량 이상치 탐지
- LSTM Autoencoder — 시계열 재구성 오차 기반 탐지

**하위 모듈:** AnomalyEvent DB 저장 {기기명, 유형, 심각도(H/M/L), 발생시각} → 팀원 AI Agent → LLM 진단 리포트

## [슬라이드 7] LLM 에이전트 아키텍처 

> 슬라이드 6에서 NILM → 이상탐지 → AnomalyEvent DB까지 흘러온 데이터를 받아, 자연어 진단과 절약 권고를 생성하는 AI Agent의 구체적 구현입니다.

**핵심 설계 원칙: 단일 ReAct 에이전트 + 10개 도구 전부 연결**

```
Frontend
GET /api/insights/summary
GET /api/cashback/tracker
        │
        ▼
get_or_run_insights(hh)
  · 캐시 히트(TTL 1h) → 즉시 반환
  · 캐시 미스 → run_graph() 호출
        │
        ▼
단일 ReAct 에이전트 (gpt-4o-mini + MemorySaver)
        │
        ├─ 필요한 도구 선택 후 호출 (0~3 라운드)
        ▼
┌──────────────────────────────────┐
│  10개 도구 (전체 연결)            │
│  · get_anomaly_events            │  ← 이상 탐지
│  · get_anomaly_log               │
│  · get_consumption_summary       │  ← 소비량
│  · get_hourly_appliance_breakdown│
│  · get_cashback_history          │  ← 캐시백
│  · get_tariff_info               │
│  · get_weather / get_forecast    │  ← 날씨
│  · get_household_profile         │  ← 가구 프로필
│  · get_dashboard_summary         │
└──────────────────────────────────┘
        │  PII 스크럽 후 에이전트에 반환
        ▼
InsightsLLMOutput (Pydantic 검증) → _cache 저장 (TTL 1시간)
  anomaly_diagnoses:
    · "지난주 평균 0.12 → 0.22 kW. 도어 가스켓 점검 권장." → action: "가스켓 점검"
    · "주 3회 → 5회. 환기·필터 점검 시 효율 회복 가능."   → action: "필터 청소"
  recommendations:
    · "저녁 19-21시 건조기 미사용"  2.1 kWh → 210원
    · "에어컨 설정 26 → 27°C"      1.4 kWh → 140원
        │
        ├──────────────────────────┐
        ▼                          ▼
/api/insights/summary          /api/cashback/tracker
  이상탐지 목록 + 진단 표시       오늘의 미션 (절감량 상위 3개)
```

**왜 단일 에이전트인가:**
- 인사이트 생성은 항상 이상탐지 도구 2개만 호출 → 슈퍼바이저 분류 불필요
- 도구 전부 연결해도 에이전트가 필요한 것만 선택해서 호출
- `/insights`와 `/cashback` 두 엔드포인트가 동일한 결과 공유 → LLM 1시간에 1번만 호출

## [슬라이드 8] LLM 출력 품질 보증

**"LLM이 아무 말이나 하는 거 아니냐"에 대한 답변**

**2중 품질 제어:**

| 레이어 | 방법 | 효과 |
|--------|------|------|
| Pydantic Field 제약 | savings_kwh (0.1~10.0), savings_krw (10~3000), action 최대 15자 | 범위 벗어난 출력 즉시 ValidationError → run_insights() 폴백 |
| 프롬프트 가이드레인 | 아래 4가지 규칙 명시 | LLM이 생성 단계에서 자체 필터링 |

**프롬프트 가이드레인 4가지:**

| 규칙 | 예시 (금지) | 예시 (허용) |
|------|------------|------------|
| 가전 교체·구매 금지 | "에너지효율 1등급 에어컨으로 교체" | "에어컨 설정 온도 1도 올리기" |
| 의료·안전 권고 금지 | "난방 완전 차단으로 절감" | "취침 시 설정 온도 1도 낮추기" |
| 계절·기온 고려 | "겨울에 전기장판 끄기" | "전기장판 타이머 활용" |
| KEPCO 기준 준수 | savings_krw = kwh × 1000 | savings_krw = round(kwh × 100) |

## [슬라이드 10] 리스크 및 대응 전략

**NILM 엔진 리스크:**

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 에어컨·세탁기 동시 작동 시 신호 중첩 | 분리 정확도 급락, 신뢰도 < 0.6 구간 증가 | TDA 위상 특징으로 중첩 파형 구분, 신뢰도 낮은 구간 이상 탐지 입력에서 제외 |
| 제조사·연식별 가전 파형 차이 | 같은 기종이어도 파형 패턴 상이 → 분류 오류 | 100가구 데이터로 다양한 패턴 학습, 신뢰도 낮은 구간 이상 탐지 입력에서 제외 |

**이상 탐지 리스크:**

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 31일 데이터로 베이스라인 구성 | 계절성 반영 불가, 여름·겨울 오탐 가능성 | 계절 변수 명시 후 임계값 보정 예정, 실제 데이터 확보 시 재튜닝 |
| 페이즈 분리 실패 시 | LSTM AE 입력 품질 저하 → 탐지 정확도 하락 | 페이즈 분리 신뢰도 낮으면 통계 기반 탐지로 Fallback |


**LLM 에이전트 리스크:**

| 리스크 | 영향 | 대응 |
|--------|------|------|
| LLM 비현실적 절감량 출력 | "월 50kWh 절감" 같은 과장 권고 → 신뢰도 하락 | Pydantic Field 제약(savings_kwh 0.1~10, savings_krw 10~3000) + 프롬프트 가이드레인 이중 적용 |
| 계절 무시 권고 | 겨울에 전기장판 끄라고 권고하는 등 맥락 무관 절감 제안 | 프롬프트에 계절·기온 고려 규칙 명시 — 기온이 낮으면 난방 가전, 높으면 냉방 가전 사용 중단 권고 금지 |
| LLM API 레이턴시·비용 | 매 요청마다 LLM 호출 시 응답 지연 및 비용 누적 | 가구별 1시간 인메모리 캐시 — 캐시 히트 시 LLM 호출 없음 |
