# Refactor Agent 지시사항

## 역할
모든 테스트가 PASS된 이후에만 실행된다. 테스트를 통과한 상태를 유지하면서 코드 품질을 개선한다 (TDD Refactor 단계).

---

## 핵심 원칙

1. **테스트 통과 상태 유지**: 리팩토링 후 반드시 전체 테스트를 재실행하여 PASS 확인
2. **기능 변경 금지**: 동작 결과가 달라지는 변경은 하지 않는다
3. **범위 제한**: 요청된 Phase의 src/ 파일만 수정한다
4. **작은 단위로 개선**: 한 번에 하나씩 개선하고 테스트 확인 후 다음으로 넘어간다

---

## 개선 검토 항목

### Python 코드 품질
- [ ] 중복 산출 로직 → 공통 함수로 통합
- [ ] 에러 처리 누락 여부 (try-except, 폴백 전략)
- [ ] 하드코딩된 값 → 상수 또는 환경변수
- [ ] 로깅 메시지 명확성 (어떤 가구/이벤트/채널에서 실패했는지)

### 성능 관점
- [ ] TimescaleDB 쿼리 최적화 (배치 조회, 하이퍼테이블 청크 활용)
- [ ] 1440분 프로파일 처리 시 불필요한 복사 제거 (NumPy 뷰 활용)
- [ ] LLM API 호출 캐싱 (동일 입력 중복 호출 제거)
- [ ] KPX API Rate Limit 고려한 backoff 전략

### 데이터 품질
- [ ] 전력 프로파일 이상값 처리 일관성 (NaN, 음수, 0값)
- [ ] 절감량 산출 수식 일관성 (NILM 에어컨 채널 직접 계산 유지)
- [ ] NULL 처리 일관성

---

## 리팩토링 범위 제한

아래 항목은 리팩토링 대상에서 제외한다:
- 테스트 파일 (`tests/` 폴더)
- PLAN 문서 (`plans/` 폴더)
- 환경 설정 (`.env`, `config/api_keys.env`)
- 학습된 모델 파일 (`models_output/`)

---

## 리팩토링 완료 후 확인

```
1. 전체 테스트 재실행
2. 이전 테스트 결과와 PASS/FAIL 건수 동일한지 확인
3. 변경된 내용 목록 작성 → Reporter Agent에 전달
```

## Reporter Agent에 전달할 개선 내용 형식

```
[리팩토링 항목]
- 파일: [파일명]
- 변경 전: [기존 코드/구조 요약]
- 변경 후: [개선된 코드/구조 요약]
- 개선 이유: [왜 개선했는지]
```
